
這份表格整理了 Simple OS 中 `task_t` 結構（PCB/TCB）的詳細欄位資訊。這對於理解系統如何保存任務現場、管理資源以及處理睡眠/排程邏輯至關重要。

### Simple OS: Task 控制區塊 (`task_t`) 欄位詳細表

| 欄位名稱 | 資料型別 | 功能說明 | 範例數值 | 邊界條件 / 限制 |
| :--- | :--- | :--- | :--- | :--- |
| **esp** | `uint32_t` | **核心堆疊指標**。當任務被切換走時，CPU 會將最後的堆疊位址存於此，以便下次切換回來時恢復現場。 | `0xC0007A20` | 必須位於核心位址空間。 |
| **pid** | `uint32_t` | **行程 ID**。系統中每個任務的唯一識別碼。 | `1` (Shell) | `0` (Kernel Main), `9999` (Idle)。 |
| **ppid** | `uint32_t` | **父行程 ID**。記錄是誰創建了這個任務（用於 `wait` 與 `exit` 鏈結）。 | `0` | 如果為 `0` 則父行程是 Kernel。 |
| **name** | `char[32]` | **任務名稱**。主要用於除錯或 `ps`/`top` 指令顯示用。 | `"shell.elf"` | 最多 31 個字元 + `\0` 結束符。 |
| **total_ticks** | `uint32_t` | **累計執行時間**。中斷發生時若該任務在 CPU 上，此數值加一。用於計算 CPU 使用率。 | `500` | 100Hz 下約 **497 天**後溢位。 |
| **page_directory** | `uint32_t` | **虛擬記憶體頁目錄 (CR3)**。任務專屬的頁表實體位址，確保行程間的記憶體隔離。 | `0x00327000` | 必須是 **實體位址** 且 4KB 對齊。 |
| **kernel_stack** | `uint32_t` | **核心堆疊頂部**。進入 Ring 0 時 TSS.esp0 指向這裡。用於防止使用者模式堆疊破壞核心。 | `0xC0008000` | 只有單個頁面大小 (4KB)。 |
| **state** | `uint32_t` | **任務目前狀態**。決定排程器是否應該挑選此任務。 | `0` (RUNNING) | `RUNNING`, `DEAD`, `WAITING`, `ZOMBIE`, `SLEEPING`。 |
| **wait_pid** | `uint32_t` | **等待的子行程 PID**。當狀態為 `TASK_WAITING` 時，記錄正在等哪一個子行程結束。 | `5` | 僅在 `sys_wait` 期間有意義。 |
| **heap_start** | `uint32_t` | **使用者堆積起點**。記錄 `malloc` 可分配空間的初始位址。 | `0x10000000` | 通常固定為 `USER_HEAP_START`。 |
| **heap_end** | `uint32_t` | **使用者堆積當前終點**。隨 `brk`/`sbrk` 系統呼叫動態移動。 | `0x10005000` | 不得超過實體記憶體極限或分頁表範圍。 |
| **cwd_lba** | `uint32_t` | **目前工作目錄 LBA**。檔案系統中該任務目前所在資料夾的硬碟位址。 | `2048` | 指向硬碟磁區編號。 |
| **wake_up_tick** | `uint32_t` | **喚醒時間點**。當任務處於 `TASK_SLEEPING` 時，排程器會檢查全域 `tick` 是否大於等於此值。 | `tick + 100` | 若為 `0` 表示不處於定時睡眠。 |
| **next** | `task_t*` | **環狀鏈結串列指標**。指向下一個任務，形成 Round-Robin 排程的基礎。 | `0x0010e1b8` | 如果只有一個任務，則指向任務自己。 |

### 💡 關鍵知識補充：PCB 與 TCB 的一體化

在你的 Simple OS 實作中，這個 `task_t` 同時扮演了兩個角色：

1.  **TCB (Task Control Block) 特性**：
    * `esp`: 保存 CPU 狀態。
    * `state`, `wake_up_tick`: 控制排程行為。
2.  **PCB (Process Control Block) 特性**：
    * `pid`, `page_directory`: 資源識別與記憶體保護。
    * `cwd_lba`: 檔案系統環境。



### 邊界安全提醒

* **堆疊溢位 (Stack Overflow)**：你的 `kernel_stack` 分配了一個 `PAGE_SIZE` (4KB)。如果 `timer_handler` 呼叫深度太深，或是局部變數太大，會直接踩到別的資料結構，導致莫名其妙的當機。
* **PID 溢位**：`next_pid` 目前是 `uint32_t` 遞增，理論上可用很久，但若長時間運作且頻繁開關進程，需要考慮 PID 回收機制，以免 PID 數字變得過大。
