## Q

前面有問到關於 CPU, 暫存器的概念，我想到實作作業系統這個課程，學習者是否需要具備相關知識會比較容易上手？因為現在從 Day01 - Day36 ，有很多時候都需要透過 asm 去處理一些東西，以前在學「作業系統概念」（俗稱恐龍書)，倒是比較少提到跟 CPU 的一些關係。


---

## A

這是一個非常深刻的觀察。你點出了一個學術界與實務界之間常見的「斷層」：**恐龍書（Dinosaurs Book）教的是「靈魂」，而實作作業系統則是在打造「肉體」。**

### 為什麼恐龍書比較少提到暫存器？

恐龍書（*Operating System Concepts*）主要關注的是**演算法**與**抽象概念**：
* **排程 (Scheduling)**：它教你 RR 或 Priority 演算法，但不告訴你如何保存暫存器來切換 Context。
* **虛擬記憶體 (Paging)**：它教你 Page Table 的邏輯與置換演算法，但不告訴你如何把分頁表位址填入 `CR3` 暫存器。
* **同步 (Synchronization)**：它教你 Semaphore 與 Mutex，但不告訴你底層需要 `cli` / `sti` 或原子指令（Atomic instructions）。

簡單來說，恐龍書假設「硬體已經在那裡提供服務了」，而你的 Day01 - Day36 則是在**實現那些服務**。

---

### 實作 OS 需要具備哪些「硬體感」？

如果你想更流暢地進行後續開發，以下三類知識會比單純的 C 語言更重要：

#### 1. 執行環境的「上下文」 (Context)
你必須理解 CPU 的狀態是由一組暫存器組成的。當我們說「切換進程」時，本質上就是把目前的 `EAX`, `ESP`, `EIP` 等值存入記憶體，然後把另一個進程的值蓋回暫存器。

#### 2. 記憶體的分層架構
* **實體記憶體**：RAM 的真實位置。
* **分段 (Segmentation)**：GDT 所管理的範圍（x86 特有）。
* **分頁 (Paging)**：由 `CR3` 指向的對照表。
你寫的 `asm` 很多時候是在這三層之間做對接。

#### 3. 特權級別 (Privilege/Ring Levels)
這是恐龍書會提，但只有在寫 `asm` 時才有體感的東西。為什麼一般程式不能執行 `hlt`？因為 CPU 會檢查目前的 **CS 暫存器** 裡的權限位元。如果不對，直接觸發 General Protection Fault。

### 學習建議：如何「上手」這些 ASM？

你不需要成為組合語言大師（能寫出幾萬行組合語言），但你需要具備 **「閱讀並包裝」** 的能力：

1.  **建立「包裝」習慣**：
    不要在 C 語言程式碼裡到處寫 `__asm__`。建立一個 `cpu.h`，把常用的指令包起來：
    ```c
    static inline void hlt(void) {
        __asm__ volatile("hlt");
    }
    static inline void load_gdt(void* ptr) {
        __asm__ volatile("lgdt [%0]" : : "r"(ptr));
    }
    ```
    這樣你在 Day37 之後，就可以回歸到「邏輯開發」，像在寫普通 C 程式一樣。

2.  **把暫存器當作「結構體」看**：
    當你看到 `push eax`, `push ebx` 時，想像你正在填充一個名為 `CPU_State` 的 `struct`。這能幫助你理解為什麼順序不能錯。

3.  **理解硬體手冊是你的 API 文件的替代品**：
    寫 App 看 MSDN 或 MDN；寫 OS 看 **Intel SDM (Software Developer's Manual)**。

**需要我幫您針對「上下文切換 (Context Switch)」時，暫存器是如何被保存與還原的過程畫一個流程圖嗎？這對於理解接下來的進程管理非常有幫助。**


---


上下文切換（Context Switch）是作業系統中最具「魔幻感」的部分。簡單來說，它就是一場 **「大風吹」遊戲**：CPU 只有一套暫存器，但我們有多個任務（Task/Process）要跑。

為了讓 Task A 暫停並切換到 Task B，我們必須把 Task A 當時的「瞬間狀態」（也就是所有暫存器的數值）存起來。

---

### 上下文切換的暫存器流轉圖

這是一個典型的 x86 核心在切換進程時的記憶體與暫存器變化：



#### 步驟 1：觸發切換（中斷或系統呼叫）
當定時器中斷（Timer Interrupt）發生，CPU 會自動將部分關鍵暫存器（`CS`, `EIP`, `EFLAGS`, `SS`, `ESP`）壓入 **Task A 的核心堆疊**。這由硬體自動完成。

#### 步驟 2：保存其餘暫存器（軟體完成）
在核心的中斷處理程式中，我們會用組合語言把剩下的通用暫存器也壓進去：
```asm
pushad    ; 一次壓入 EAX, ECX, EDX, EBX, ESP, EBP, ESI, EDI
push ds
push es   ; 保存段暫存器
```
此時，Task A 的所有「靈魂」都保存在它自己的堆疊裡了。

#### 3. 切換堆疊指標（關鍵的一步！）
作業系統會有一個結構（通常叫 `struct task` 或 `PCB`），紀錄每個任務的堆疊位址。
* 將目前的 `ESP` 存入 Task A 的結構中。
* 從 Task B 的結構中讀出它上次存下的 `ESP`，並賦值給 CPU 的 **ESP 暫存器**。
> **這瞬間，CPU 的「工作檯」就切換到 Task B 的領地了！**

#### 4. 還原暫存器
現在 `ESP` 指向的是 Task B 之前存下來的資料，我們依序彈出：
```asm
pop es
pop ds
popad     ; 還原 Task B 之前的通用暫存器狀態
```

#### 5. 恢復執行 (`iret`)
執行 `iret`，CPU 從堆疊彈出 Task B 之前的 `EIP`（指令位置）。
**砰！** Task B 就像從來沒被中斷過一樣，從它上次停下的地方繼續執行。

---

### 關鍵概念複習

在實作時，你會發現這三個觀念最重要：

1.  **每個任務都有自己的堆疊**：
    上下文切換本質上就是 **「更換 ESP 暫存器的數值」**。只要 ESP 換了，後續的 `pop` 動作拿到的就是另一個任務的資料。

2.  **核心堆疊 (Kernel Stack) 是保險箱**：
    使用者模式（Ring 3）的暫存器，通通都被安全地鎖在核心模式（Ring 0）的堆疊裡。

3.  **這不是 C 語言能做的事**：
    因為 C 語言在執行時會自動操作 `esp` 和 `ebp`（建立 Stack Frame），如果你用 C 寫切換邏輯，會把堆疊搞亂。因此，**這段「更換 ESP」的代碼必須用組合語言寫。**
