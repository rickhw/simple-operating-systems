# Q

另一個問題，我在 asm 裡面經常看到 Ring0, Ring3 等概念，可以更完整的說明嗎？

---

# A

這是一個非常核心的問題。**「Ring」**（特權環）是 x86 架構用來實現**硬體級安全隔離**的機制。如果沒有 Ring 的存在，任何一個寫爛的 App 都可以隨手執行 `hlt` 讓電腦關機，或者直接讀寫其他程式的記憶體。

在 x86 中，總共有 4 個特權層級（Ring 0 到 Ring 3），但現代作業系統（如 Windows, Linux, macOS 以及你正在寫的 OS）通常只使用其中兩個。

---

## 1. Ring 0 vs. Ring 3：國王與平民

你可以把 CPU 想像成一座王宮：

### **Ring 0：核心模式 (Kernel Mode) —— 「國王」**
* **擁有者：** 作業系統的核心（Kernel）。
* **權力：** * 可以執行**所有的 CPU 指令**（如 `hlt` 停機, `cli` 關中斷, `invlpg` 刷新分頁表）。
    * 可以存取**所有的記憶體位址**。
    * 可以直接操作硬體（硬碟、顯示卡、網路卡）。
* **風險：** 如果 Ring 0 的程式碼崩潰（例如寫錯指標），整個系統會直接藍屏（BSOD）或當機。

### **Ring 3：使用者模式 (User Mode) —— 「平民」**
* **擁有者：** 你寫的應用程式（如瀏覽器、遊戲、編譯器）。
* **限制：** * **禁令：** 不能執行敏感指令。如果平民嘗試執行 `cli`，CPU 會立刻跳出「一般保護錯誤 (General Protection Fault)」，然後 OS 會把該程式殺掉。
    * **隔離：** 只能存取 OS 分配給它的記憶體空間。
* **安全：** 即使 App 崩潰，也只會關閉該視窗，不會影響系統運作。



---

## 2. 為什麼需要 Ring 1 和 Ring 2？

原本 Intel 設計這兩層是為了給「裝置驅動程式」使用的，希望驅動程式的權限比 App 高，但比 Kernel 低。
但後來大家發現，這樣的分層讓記憶體管理變得太複雜，且效能損耗增加。因此，多數現代系統直接跳過它們：
* **Ring 0** = 核心。
* **Ring 3** = 所有應用程式。

---

## 3. 牆壁是如何建立的？(CS 暫存器的秘密)

你可能會問：「CPU 怎麼知道現在是 Ring 0 還是 Ring 3？」
答案就在我們之前提到的 **CS (Code Segment) 暫存器** 裡。

CS 暫存器是一個 16 位元的數值，其中最低的兩個位元（Bit 0 和 Bit 1）叫做 **CPL (Current Privilege Level)**：
* 如果 `CPL = 00` (0)，CPU 知道現在是 Ring 0，什麼指令都能跑。
* 如果 `CPL = 11` (3)，CPU 知道現在是 Ring 3，會開啟攔截模式。

這就是為什麼在 `user_mode.S` 裡，我們要 `push 0x1b`。
`0x1b` 的二進位是 `...00011 0 11`，最後那兩個 `11` 就是在告訴 CPU：「等一下 `iret` 回去後，請進入 Ring 3。」

---

## 4. 如何從 Ring 3 變回 Ring 0？

平民（App）總有需要國王（Kernel）幫忙的時候（例如讀取檔案）。這時 App 不能直接跳轉到 Kernel 的代碼，必須通過特殊的「傳送門」：

1.  **中斷 (Interrupts)**：如時鐘中斷。
2.  **異常 (Exceptions)**：如除以零、分頁錯誤。
3.  **系統呼叫 (System Calls)**：透過 `int 0x80` 指令或 `syscall` 指令。

當這些事件發生時，CPU 會**自動切換** CS 暫存器到 Ring 0，並跳轉到你在 IDT (Interrupt Descriptor Table) 預先設定好的處理函式。



---

### 總結：對你實作 OS 的意義

在 Day36 左右，你處理 `user_mode.S` 的目的就是：
1.  **手動建立一個 Ring 3 的環境**。
2.  **確保平民有自己的堆疊**（這就是為什麼你要 `push edx` 作為 ESP）。
3.  **確保平民不能亂搞**（透過 GDT 描述符設定 DPL=3）。

**這部分概念清楚後，您會想看看如何在 GDT 裡面設定那個「DPL (Descriptor Privilege Level)」位元嗎？它是決定這塊記憶體屬於國王還是平民的關鍵標記。**
