看到這個充滿紅字與 Exception 的 QEMU 崩潰畫面，千萬別氣餒！這是所有 OS 開發者都會經歷的「成年禮」。

我們來當一次系統法醫，從你提供的 QEMU Dump 裡尋找線索：
你看這行 `v=0e e=0002 i=0 cpl=3 IP=001b:0804806c pc=0804806c SP=0023:00108f2c CR2=fffffffc`

這裡面隱藏了三個核彈級的線索：

1. **`v=0e`**：Exception 14，也就是 **Page Fault (分頁錯誤)**。
2. **`cpl=3`**：案發時 CPU 正在 **Ring 3 (User App)** 運行。
3. **`SP=0023:00108f2c`**：這是最致命的證據！**Ring 3 的堆疊指標 (SP) 居然是 `00108f2c`！**

`0x00108f2c` 是什麼地方？這正是我們在 `boot.S` 裡宣告的 **Kernel Stack (核心堆疊)** 的位置！
Ring 3 的應用程式，竟然拿到了上帝 (Ring 0) 的堆疊。當你的 `app.c` 試圖宣告 `char buffer[128];` 或是呼叫 `sys_read` 時，CPU 試圖把資料寫進這個位址。但分頁機制 (Paging) 發現這是核心領土，Ring 3 無權寫入，於是當場引爆 Page Fault，最後導致連環車禍（Double Fault）重開機。

### 🛠️ 終極修復包：阻斷編譯器作祟與修復虛實縫合

這是一個由「參數傳遞錯位」與「GCC 編譯器暗中作祟」聯手造成的 Bug。請跟著我執行以下三步：

#### 1. 馴服強勢的 GCC 編譯器 (`build-app.sh`)

Debian 環境下的 GCC 預設會開啟 PIE (Position Independent Executable) 和 Stack Protector (堆疊防護)。這會讓 GCC 在你的 `app.c` 裡偷偷塞入存取 `gs:0x14` 或 GOT 表的機器碼，這在沒有標準函式庫的自製 OS 裡絕對會死機。

請修改你的 **`build-app.sh`**，加上這三個關鍵的解除武裝參數：`-fno-pie -fno-pic -fno-stack-protector`

```bash
# 1. 編譯獨立的 User App (加入解除武裝的參數)
docker run --platform linux/amd64 --rm -v $(pwd):/workspace -w /workspace os-builder gcc -m32 -ffreestanding -nostdlib -fno-pie -fno-pic -fno-stack-protector -c app.c -o app.o

# 2. 連結 App
docker run --platform linux/amd64 --rm -v $(pwd):/workspace -w /workspace os-builder ld -m elf_i386 -Ttext 0x08048000 app.o -o my_app.elf

```

#### 2. 打造無懈可擊的 `enter_user_mode` (`asm/user_mode.S`)

前幾天我給的組合語言可能因為沒有加上標準的 `push ebp`，導致堆疊參數錯位，拿到了舊的 Kernel ESP。我們換上這套最嚴謹的寫法：

請修改 **`asm/user_mode.S`**：

```nasm
global enter_user_mode
enter_user_mode:
    ; 標準的 C 語言函式開頭
    push ebp
    mov ebp, esp
    
    ; 因為 push 了 ebp，參數的位移要 +4
    mov eax, [ebp+8]  ; 參數 1: entry_point
    mov edx, [ebp+12] ; 參數 2: user_stack_ptr

    mov cx, 0x23
    mov ds, cx
    mov es, cx
    mov fs, cx
    mov gs, cx

    push 0x23         ; SS (User Data Segment)
    push edx          ; ESP (給平民專用的堆疊！)
    pushfd
    pop ebx
    or ebx, 0x200     ; 開啟中斷
    push ebx
    push 0x1b         ; CS (User Code Segment)
    push eax          ; EIP
    iret

```

#### 3. 確保 Kernel 真的有分配並傳遞堆疊 (`lib/kernel.c`)

請打開 **`lib/kernel.c`**，檢查你啟動 App 的那一段邏輯，**確保你有分配 4KB 的記憶體，並且把它當作第二個參數傳給 `enter_user_mode**`：

```c
        kprintf("\nLoading ELF Application...\n");
        uint32_t entry_point = elf_load(real_app);
        
        if (entry_point != 0) {
            kprintf("Allocating User Stack and Dropping to Ring 3...\n\n");
            
            // 1. 給平民一塊專屬的 4KB 記憶體當堆疊
            uint32_t user_stack_phys = pmm_alloc_page();
            map_page(0x083FF000, user_stack_phys, 7);
            uint32_t user_stack_ptr = 0x083FF000 + 4096;
            
            // 2. 呼叫我們更新版的跳轉函式 (必須傳遞兩個參數！)
            extern void enter_user_mode(uint32_t entry_point, uint32_t user_stack);
            enter_user_mode(entry_point, user_stack_ptr);
        }

```

---

完成這三步後，請先跑一次 `./build-app.sh` 確保 App 用新的參數重新編譯，接著再跑一次 `make clean && make run`。

這一次，你的 Ring 3 應用程式將擁有自己專屬的、位於 `0x083FFXXX` 的完美堆疊，再也不會誤觸神明（Kernel）的領土了！快試試看，我相信這次絕對能看見 `hello.txt` 的內容印在畫面上！
