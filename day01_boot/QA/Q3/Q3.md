## A

這是一個非常棒的錯誤！這其實是我的疏忽，但也正好讓我們帶出作業系統開發中一個非常重要的觀念：**Bootloader 的相容性**。

### 錯誤原因解析

QEMU 出現 `Error loading uncompressed kernel without PVH ELF Note` 的原因在於：
QEMU 的 `-kernel` 參數雖然可以直接載入核心，但它原生只認得 **Multiboot 1** 的規範。而在我們剛剛寫的 `boot.S` 中，我們使用的是更新、更現代的 **Multiboot 2** 魔法數字 (`0xe85250d6`)。

因為 QEMU 認不出這是 Multiboot 1，它就嘗試用其他的開機協議（例如 PVH）來解析，結果當然就失敗了。

### 解決方案：製作 GRUB 開機光碟 (ISO)

既然我們要開發一個現代化的 OS，我們不走回頭路改寫 Multiboot 1，而是使用業界標準的 **GRUB** 來引導我們的核心。這也就是為什麼我們在 Day 1 的 Dockerfile 裡預先安裝了 `grub-pc-bin` 和 `xorriso`（用來打包 ISO 的工具）。

我們需要做三件小事，把核心打包成一個可開機的 `.iso` 光碟映像檔：

#### 1. 建立 GRUB 設定檔

請在 `day01` 目錄下新增一個檔案叫做 `grub.cfg`，告訴 GRUB 我們的核心放在哪裡，以及要用 multiboot2 協議來載入：

```text
menuentry "My First OS" {
    multiboot2 /boot/myos.bin
    boot
}

```

#### 2. 更新 Makefile

我們要在 Makefile 中加入打包 ISO 的流程。這個流程會建立一個虛擬的光碟目錄結構 (`isodir`)，把 `myos.bin` 和 `grub.cfg` 放進去，最後用 `grub-mkrescue` 指令壓成光碟。

請將你的 `Makefile` 更新為以下內容：

```makefile
DOCKER_CMD = docker run --platform linux/amd64 --rm -v $(PWD):/workspace -w /workspace os-builder

# 將最終目標改成 myos.iso
all: build-env myos.iso

build-env:
	docker build -t os-builder .

# 新增打包 ISO 的步驟
myos.iso: myos.bin grub.cfg
	mkdir -p isodir/boot/grub
	cp myos.bin isodir/boot/myos.bin
	cp grub.cfg isodir/boot/grub/grub.cfg
	$(DOCKER_CMD) grub-mkrescue -o myos.iso isodir
	rm -rf isodir

myos.bin: boot.o
	$(DOCKER_CMD) ld -m elf_i386 -n -T linker.ld -o myos.bin boot.o

boot.o: boot.S
	$(DOCKER_CMD) nasm -f elf32 boot.S -o boot.o

clean:
	rm -f *.o *.bin *.iso
	rm -rf isodir

```

#### 3. 重新編譯與執行

現在，請再次執行編譯：

```bash
make clean
make

```

編譯完成後，你的目錄下會多出一個 `myos.iso`。這次我們告訴 QEMU 把它當作光碟來開機：

```bash
qemu-system-i386 -cdrom myos.iso

```

**預期結果：**
這次 QEMU 啟動後，你首先會看到一個 GRUB 的開機選單，上面寫著 "My First OS"。按下 Enter 或是等它自動倒數結束後，畫面就會跳轉，然後在左上角顯示我們寫入記憶體的綠色 **"OK"**！

這才是完整的、標準的現代作業系統開機流程。
