### 🔍 案發現場解密

請看你貼的 Log 裡面的這幾個關鍵資訊：

1. **死因：** `check_exception old: 0xffffffff new 0xd`
* `0xd` 換算成十進位是 13。在 x86 架構中，13 號中斷叫做 **General Protection Fault (一般保護錯誤, #GP)**。CPU 遇到了嚴重的權限或記憶體違規！


2. **兇器：** `e=0028`
* 這是錯誤代碼 (Error Code)。`0x28` 剛好就是我們 Day 17 準備要塞給 TSS 的 GDT 區段選擇子 (Selector)！


3. **案發瞬間的暫存器：**
* `EAX=0010002b`：你看 EAX 的低 16 位元是不是剛好是 `0x002B`？這完美吻合了我們在 `tss_flush` 裡寫的 `mov ax, 0x2b`。


4. **【致命的鐵證】：** `GDT=     00107080 00000017`
* 這行是 GDT 暫存器的狀態。前面是記憶體位址，**後面的 `00000017` 是 Limit (長度限制)**。
* `0x17` 換算成十進位是 **23**。
* 為什麼是 23？因為 `(3個元素 * 8 bytes) - 1 = 23`。



### 💡 破案總結

你一定是在更新 `lib/gdt.c` 的時候，把下面新增的 `gdt_set_gate` 都貼上去了，但**忘記修改 GDT 陣列的長度限制**了！

CPU 眼中的 GDT 依然只有 Day 6 時代的「3 個」項目。當 `tss_flush` 執行 `ltr ax` 試圖去讀取第 5 個項目 (`0x28`) 時，CPU 檢查 Limit 發現越界了，立刻拋出 #GP (13) 錯誤。
而此時 `IDT= 00000000 00000000` (IDT 還沒初始化)，CPU 找不到處理中斷的跳板，於是引發 Double Fault (8) -> Triple Fault -> 無限重啟！

---

### 🛠️ 解決方案

請打開 **`lib/gdt.c`**，檢查這兩個地方是否有正確更新成 **6**：

1. **陣列宣告是否擴充了？**
```c
// 原本可能是 gdt_entry_t gdt_entries[3];
gdt_entry_t gdt_entries[6]; 

```


2. **`init_gdt` 裡面的 Limit 是否更新了？**
```c
void init_gdt(void) {
    // [就是這裡！] 原本是 * 3，必須改成 * 6
    gdt_ptr.limit = (sizeof(gdt_entry_t) * 6) - 1; 
    gdt_ptr.base  = (uint32_t)&gdt_entries;
    // ... 下面的 gdt_set_gate ...
}

```



把這兩個 `3` 改成 `6` 之後，重新 `make clean && make`，GDT 的長度就會變成 `0x2F` (47)，`ltr` 指令就能順利通關，成功降落到 User Mode 了！
