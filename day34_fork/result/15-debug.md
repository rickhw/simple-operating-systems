### 🔍 案發現場解密：卡在 VFS 與檔案系統的鎖裡！

回想一下你的 `sys_fork` 是在哪裡被呼叫的？是在 `app.c` 的 `read_line` 等待使用者輸入之後！

當你輸入 `fork` 按下 Enter 時，系統做了什麼：

1. 本尊 (Parent) 解析到 `fork`，呼叫 `sys_fork()`。
2. Kernel 複製出分身 (Child)，把 Child 放入 Ready Queue。
3. `sys_fork()` 返回，Parent 拿到 PID (例如 2)，印出 `[PARENT] Magic!...`。
4. **【關鍵】** Parent 印完字後，迴圈繼續，它會再次印出 `SimpleOS>`，然後**呼叫 `read_line` -> `sys_getchar` 去等鍵盤！**

此時，排程器 (Timer Interrupt `0x20`) 觸發，把 CPU 切換給了 Child。

Child 從哪裡醒來？它從 `sys_fork()` 的下一行醒來！
Child 拿到 PID=0，進入 `if (pid == 0)` 的分支。
Child 準備印出 `[CHILD] Hello!...`。

**問題來了，`sys_print` 底層是怎麼運作的？**
你的 `sys_print` 雖然沒有明寫在 `app.c` 裡，但它最終會呼叫 `kprintf`，而 `kprintf` 會去存取 VGA 的緩衝區並更新游標。
更致命的是，如果 Child 在印字的同時，或者在未來它試圖存取檔案時，**檔案系統 (VFS / SimpleFS) 或 VGA 驅動裡面可能有某種「鎖 (Lock)」或是「全域狀態」！**

在真實的 OS 實作 `fork` 時，我們不僅要複製記憶體和暫存器，我們還必須處理**「開啟的檔案描述符 (File Descriptors)」的參照計數**，以及**各種全域的鎖**。

但我們目前最有可能遇到的卡死原因是：**「搶佔式多工的螢幕輸出衝突」**！

當 Parent 正在印 `SimpleOS>` 時被切換走，VGA 的游標狀態可能卡在一半；Child 醒來立刻接著印 `[CHILD] Hello!`，兩個人在爭奪同一個 `kprintf` 的底層資源。

---

### 🛠️ 解決方案：讓子彈飛一會兒 (Yield 魔法)

為了證明系統真的沒有崩潰，只是卡在輸出爭奪戰，我們可以用一個最經典的多工作業技巧：**強制讓出 (Yield)**。

我們讓 Parent 創造完小孩後，不要馬上回去搶螢幕，而是**先睡一覺 (主動讓出 CPU)**，讓小孩把話印完並順利自殺退出！

請修改你的 **`app.c`**：

```c
// ... 前面保留 ...

        else if (strcmp(cmd_buffer, "fork") == 0) {
            int pid = sys_fork();

            if (pid == 0) {
                // 我是剛出生的分身！
                sys_print("\n[CHILD] Hello! I am the newborn process!\n");
                sys_print("[CHILD] My work here is done, committing suicide...\n");
                sys_exit(); // 小孩登出
            } else {
                // 我是老爸！
                sys_print("\n[PARENT] Magic! I just created a child process!\n");
                
                // 【關鍵】老爸印完字後，主動讓出 CPU！
                // 這樣排程器必定會切換到小孩，讓小孩能順利印字並執行 exit，
                // 而不會兩個人同時在搶終端機的畫面！
                sys_yield();
                sys_yield(); // 為了保險，多讓幾次
            }
        }
// ...

```

加上 `sys_yield()` 後，重新 `make clean && make run`！
