### 1. 這些標準函式庫到底該放在哪裡？

首先，稍微釐清一個重要觀念：雖然我們在 OS 開發環境中禁用了標準函式庫 (`-ffreestanding`)，但你列出的這四個：`<stdint.h>`, `<stddef.h>`, `<stdarg.h>`, `<stdbool.h>` 是**編譯器 (GCC) 內建提供**的，不依賴底層 OS，所以我們可以安心使用。

在 C 語言的專案架構中，有一個黃金守則：**「誰需要，誰就引入；如果 `.h` 檔的宣告需要，就放在 `.h`；如果只有 `.c` 檔的實作需要，就放在 `.c`。」**

以下是具體的配置建議：

* **`stdint.h` (提供 `uint8_t`, `uint32_t` 等) & `stddef.h` (提供 `size_t`)**
* **應該放在：** 幾乎所有的 `.h` 檔（如 `tty.h`, `utils.h`, `gdt.h`）。
* **原因：** 因為你的函式宣告（例如 `void* memcpy(void* dst, const void* src, size_t size);`）直接用到了 `size_t`。別人 `#include "utils.h"` 時，編譯器必須先認識 `size_t` 是什麼。


* **`stdarg.h` (提供 `va_list` 處理不定參數)**
* **應該放在：** `utils.h` 與 `utils.c`。
* **原因：** `utils.h` 裡的 `kprintf(const char* format, ...)` 宣告，以及 `utils.c` 裡的實作都需要它。


* **`stdbool.h` (提供 `bool`, `true`, `false`)**
* **應該放在：** `utils.c`。
* **原因：** 我們只有在 `itoa` 這個函式的「內部實作」用到 `is_negative` 旗標，函式對外宣告的介面並沒有用到 `bool`。把它藏在 `.c` 裡可以保持標頭檔的乾淨。

### 2. 循環依賴 (Circular Dependency) 在 C 語言中 OK 嗎？

你注意到了 `utils.c` 需要 `#include "tty.h"` (為了用 `terminal_putchar` 印字)，而 `tty.c` 又需要 `#include "utils.h"` (為了用 `memcpy` 滾動螢幕)。

**結論是：這樣是完全 OK 的，這在 C 語言中非常常見！**

但在這裡，我們必須嚴格區分兩種狀況：

#### 🟢 安全的互相依賴：`.c` 互相 include 對方的 `.h`

你目前的情況就是這種。`utils.c` 知道 `tty.h` 提供了什麼工具，`tty.c` 也知道 `utils.h` 提供了什麼工具。在編譯期，GCC 是「一個一個檔案獨立編譯」的：

1. 編譯 `utils.c` 時，編譯器看到 `tty.h` 的宣告，說：「好，我知道有 `terminal_putchar` 這個東西，實作晚點 Linker 會去找。」
2. 編譯 `tty.c` 時，編譯器看到 `utils.h` 的宣告，說：「好，我知道有 `memcpy`，實作晚點找。」
這兩者並不會產生衝突，最後在 `ld` 連結階段，Linker 會完美地把它們縫合在一起。

#### 🔴 災難的循環依賴：`.h` 互相 include 對方的 `.h`

如果今天是 `utils.h` 裡面寫了 `#include "tty.h"`，然後 `tty.h` 裡面又寫了 `#include "utils.h"`，這就會引發災難。
C 語言的 `#include` 是非常暴力的「文字複製貼上」。這會導致編譯器陷入無限輪迴的複製貼上，直到記憶體爆炸。

**防禦機制：Include Guards (標頭檔防衛)**
這就是為什麼我們要在每一個 `.h` 檔的開頭跟結尾加上這三行保命符：

```c
#ifndef TTY_H
#define TTY_H
// 你的宣告...
#endif

```

這確保了即使某個檔案不小心被 `#include` 了幾百次，編譯器也只會展開它一次。
