
# Q1: GDT 跟 Day2 boot.S 提到的 16K 應該是不一樣的東西？

沒錯，你的直覺非常敏銳！它們是**完全不同**的東西，但在系統底層的運作上卻是「互相依賴」的關係。

這是一個非常棒的提問，把這兩個概念拆解清楚，對理解作業系統的記憶體管理會有極大的幫助。讓我用一個具體的比喻來為你說明：

### 1. 16KB 堆疊 (Stack)：你的「專屬工作桌」

在 Day 2 裡，我們在 `.bss` 區段用 `resb 16384` 挖出來的那塊空間，是**實實在在用來裝資料的記憶體容器**。

* **角色：** 它是 CPU 執行 C 語言函式時的「工作桌」。
* **用途：** 當你呼叫函式、宣告區域變數 (`int x = 10;`)、或者傳遞參數時，資料就會被放在這張桌子上。函式執行完，東西就清空。它負責的是程式執行的**「動態狀態」**。

### 2. GDT (全域描述符表)：大樓的「安管權限名冊」

Day 6 我們實作的 GDT，本身不是用來裝變數或一般資料的。它是一張**設定表**，是用來給 CPU 的硬體防護機制（MMU）看的。

* **角色：** 它是系統的「安全與分區規則」。
* **用途：** 它告訴 CPU：「從位址 0 到 4GB 這一整塊地，是屬於『Data Segment (資料區段)』，並且只有最高權限 (Ring 0) 的核心程式可以讀寫。」

---

### 它們的交集：為什麼我們兩個都需要？

這兩者的關係是：**16KB 的堆疊，物理上是「座落於」GDT 所定義的「資料區段 (Data Segment)」範圍之內。**

在我們寫作業系統的過程中，這兩件事的發生順序很有趣：

1. **Day 2 借用權限：** 我們宣告了 16KB 的工作桌，並且順利使用了。為什麼那時沒有 GDT 也能跑？因為 GRUB (Bootloader) 在把控制權交給我們之前，已經「好心」幫我們設定了一個暫存的 GDT，讓 CPU 允許我們讀寫記憶體。
2. **Day 6 宣示主權：** 雖然 GRUB 給的 GDT 可以用，但作為一個成熟的作業系統，我們不能永遠依賴別人的設定。所以我們自己寫了一份 GDT（安管名冊），並透過 `lgdt` 指令載入 CPU，正式覆蓋掉 GRUB 的設定，完全掌控這棟大樓的存取規則。

如果我們今天把 GDT 裡 Data Segment 的權限設定為「唯讀」，那麼當 CPU 下一步試圖把變數放進那個 16KB 的堆疊時，硬體就會立刻觸發崩潰（Protection Fault），因為安管規則不允許寫入。

---

簡單來說：**GDT 定義了「哪裡可以放東西、誰有資格放」，而 16KB 的堆疊就是那個「實際拿來放東西的地方」。**
