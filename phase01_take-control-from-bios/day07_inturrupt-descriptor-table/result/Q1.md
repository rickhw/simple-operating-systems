
# Q1: 我跑起來會出現無限的 重開機，抓圖抓到的狀況是這樣，有可能要檢查哪裡？

恭喜你，你剛剛觸發了作業系統開發中最經典、也最讓人又愛又恨的現象：**Triple Fault（三重錯誤）導致的無限重啟！**

看到 QEMU 一直在 SeaBIOS 和 GRUB 之間瘋狂循環，這代表你的核心在開機的極早期階段（甚至在來得及印出錯誤訊息之前）就已經崩潰了。

別擔心，你的程式碼邏輯可能**完全沒有寫錯**，這通常是 C 語言編譯器（GCC）跟我們開的一個巨大玩笑！

### 🔍 兇手調查：編譯器的「自作聰明」與三重錯誤

在 Day 7 的測試程式碼中，我們寫了這段：

```c
int a = 10;
int b = 0;
int c = a / b; 

```

因為我們在 `Makefile` 裡開啟了 `-O2` 最佳化，GCC 在編譯時非常聰明，它會「預先計算」這些固定的數字。當它看到 `10 / 0` 時，它知道這是非法的（Undefined Behavior）。

為了懲罰這種行為，GCC **不會**幫你產生真實的「除法運算（`div`）」機器碼，而是直接塞入一個名為 `ud2`（Undefined Instruction，未定義指令）的毒藥指令，用來強制讓程式崩潰。

接下來，CPU 的連鎖反應（Triple Fault）就發生了：

1. **第一重：** CPU 執行到 `ud2`，這不是除以零！這會觸發 **Exception #6 (Invalid Opcode, 無效指令)**。
2. **第二重：** CPU 去查你的 IDT 聯絡簿，發現第 6 號中斷是空的（我們只設定了第 0 號），於是 CPU 覺得事情大條了，觸發 **Exception #8 (Double Fault, 雙重錯誤)**。
3. **第三重：** CPU 再次去查 IDT，發現第 8 號中斷也是空的！CPU 崩潰，宣告放棄治療，直接發出 **Triple Fault** 訊號，迫使主機板斷電重置 ➡️ **無限重啟**！

---

### 🛠️ 解決方案：騙過編譯器

我們必須讓 GCC 乖乖產生「真實的除法機器碼」，或者直接用組合語言呼叫中斷。請打開你的 `kernel.c`，把測試的那段改成以下兩種方式的**其中一種**：

#### 方法 A：使用 `volatile` 關鍵字（推薦）

加上 `volatile` 是警告編譯器：「這個變數的值可能會隨時改變，不准你做任何最佳化，給我老老實實地在執行期間去算它！」

```c
    kprintf("Testing Divide by Zero Exception...\n");
    int a = 10;
    volatile int b = 0; // [關鍵修改] 加上 volatile
    int c = a / b;      // 這次 CPU 就會真的執行除法，並觸發 0 號中斷了！
    kprintf("Result: %d\n", c);

```

#### 方法 B：直接用組合語言呼叫（最暴力直接）

我們可以直接發送軟體中斷 0 給 CPU，假裝發生了除以零。

```c
    kprintf("Testing Divide by Zero Exception...\n");
    // 直接觸發 0 號中斷
    __asm__ volatile ("int $0");
    kprintf("This line will never be reached.\n");

```

---

### 再次驗證

請修改 `kernel.c` 後，執行：

```bash
make clean
make
qemu-system-i386 -cdrom myos.iso

```

這次，你應該就能成功看到 QEMU 畫面停下來，並印出我們設定的紅色（或你設定的顏色）警報：
`[KERNEL PANIC] Exception 0: Divide by Zero!`
`System Halted.`

**趕快去試試看能不能打破這個無限輪迴！這是一個非常寶貴的除錯經驗，證明了在底層開發中，我們不僅要對抗硬體，還要跟編譯器鬥智。成功攔截錯誤後跟我說！**
