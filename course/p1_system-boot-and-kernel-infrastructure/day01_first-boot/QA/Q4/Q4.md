
# Q4: 跑起來出現 GRUB 開機選單，按下 enter 後沒有出現 綠色的 OK

## A:

看到這些截圖，恭喜你！你已經成功讓 GRUB 載入，這代表光碟映像檔的製作和環境配置都完全正確了！

至於這個錯誤 `error: unsupported tag: 0x5c7`，這是一個非常經典、也非常有教育意義的錯誤。踩到這個坑真的太好了，這讓我們有機會深入了解作業系統與 Bootloader 之間的「契約」。

### 錯誤原因解析：GRUB 被騙了

Multiboot2 規範非常嚴謹。我們在 `boot.S` 的開頭定義了「魔法數字」和「標頭長度」，這等於是告訴 GRUB：「嘿，我是相容 Multiboot2 的作業系統喔！」

然而，Multiboot2 規定在這些基本資訊之後，可以接上一系列的「標籤 (Tags)」來要求 GRUB 提供額外資訊（例如記憶體分佈、FrameBuffer 等）。重點來了：**就算你什麼標籤都不要求，你也必須給它一個「結束標籤 (End Tag)」，告訴 GRUB 標頭到此為止。**

因為我們昨天寫的 `boot.S` 少了這個「結束標籤」，GRUB 在讀完檢查碼之後，就繼續往下讀取我們的 `.text` 區段（也就是實際執行的程式碼 `mov dword [0xb8000]...`），並把這些機器碼誤認為是 Multiboot2 的標籤來解析。結果它解析出了一個不存在的標籤 `0x5c7`，於是生氣地吐出錯誤並拒絕載入核心。

---

### 解決方案：補上結束標籤

我們只需要在 `boot.S` 的標頭區塊中，補上一個標準長度為 8 bytes、內容全為 0 的結束標籤即可。

請修改你的 `day01/boot.S`，將 `section .multiboot` 的部分更新成這樣（注意 `header_end` 標籤之前新增的幾行）：

```assembly
section .multiboot
align 8
header_start:
    dd 0xe85250d6                ; Multiboot2 魔法數字
    dd 0                         ; 架構 0 (代表 i386)
    dd header_end - header_start ; 標頭總長度
    ; 檢查碼 (Checksum) = -(魔法數字 + 架構 + 長度)
    dd 0x100000000 - (0xe85250d6 + 0 + (header_end - header_start)) 

    ; === 這裡是新增的結束標籤 (End tag) ===
    align 8
    dw 0    ; type = 0
    dw 0    ; flags = 0
    dd 8    ; size = 8
header_end:

section .text
global _start
_start:
    ; VGA 文字模式的記憶體起始位置是 0xb8000
    mov dword [0xb8000], 0x2f4b2f4f 
    
    ; 關閉硬體中斷
    cli 

.hang:
    ; 讓 CPU 進入休眠狀態
    hlt
    jmp .hang

```

---

### 再次編譯與測試

修改存檔後，請再次執行熟悉的三部曲：

```bash
make clean
make
qemu-system-i386 -cdrom myos.iso

```

**預期結果：**
這次按下 Enter 後，GRUB 就能正確解析標頭，把控制權交給我們的 `_start` 函數。你應該會看到黑色的背景上，左上角出現閃亮亮的綠色 **"OK"**！
