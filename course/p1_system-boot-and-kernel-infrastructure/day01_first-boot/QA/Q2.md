## A

太好了！Docker image 成功編譯出來了，這代表我們已經順利拿到 x86_64 的環境。不過，我們遇到了第二個常見的跨平台編譯問題：**Linker (連結器) 的架構衝突**。

### 錯誤原因解析

錯誤訊息 `ld: i386 architecture of input file 'boot.o' is incompatible with i386:x86-64 output` 說得很清楚：

1. `boot.S` 被 `nasm` 編譯成了 32 位元的格式 (`-f elf32`)。這是對的，因為 Multiboot2 規範在啟動時會將 CPU 處於 32 位元保護模式。
2. 但是，因為我們的 Docker 環境是 64 位元 (`amd64`)，裡面的 `ld` 預設會嘗試輸出 64 位元的執行檔 (`x86-64`)。
3. 32 位元的輸入配上 64 位元的輸出，`ld` 就罷工了。

另外，你可能也有注意到一個警告：`WARNING: The requested image's platform...`。這是因為在 `docker run` 時，我們也需要明確告訴 Docker 使用 `amd64` 平台運行，否則 M1 的 Docker 會碎碎念。

### 解決方案：更新 Makefile

我們只需要在 `Makefile` 中做兩個小調整：

1. 在 `DOCKER_CMD` 加上 `--platform linux/amd64` 消除警告。
2. 在 `ld` 指令加上 `-m elf_i386`，強制連結器輸出 32 位元的格式。

請將你的 `day01/Makefile` 更新為以下內容：

```makefile
# 加入 --platform linux/amd64 消除執行時的架構警告
DOCKER_CMD = docker run --platform linux/amd64 --rm -v $(PWD):/workspace -w /workspace os-builder

all: build-env myos.bin

build-env:
	docker build -t os-builder .

myos.bin: boot.o
	# 加入 -m elf_i386 強制以 32 位元格式連結
	$(DOCKER_CMD) ld -m elf_i386 -n -T linker.ld -o myos.bin boot.o

boot.o: boot.S
	$(DOCKER_CMD) nasm -f elf32 boot.S -o boot.o

clean:
	rm -f *.o *.bin

```

### 再次編譯與測試

修改完成後，請再次執行：

```bash
make clean
make

```

這次編譯應該就會完全順利，不會有任何錯誤或警告了。接著，就是見證奇蹟的時刻，請執行：

```bash
qemu-system-i386 -kernel myos.bin

```
