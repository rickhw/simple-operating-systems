
# Q1: 看起來沒什麼問題，可以順利看到預期結果。好奇問一下，boot.S 裡面有一塊 16KB 記憶體空間作為堆疊，這句話是什麼意思？可以更近一步解釋嗎？

這是一個非常核心且專業的好問題！能注意到這個細節，代表你完全抓住了底層系統運作的關鍵。

在平常寫高階語言（Java、Python 甚至是跑在 Linux 上的 C）時，我們把「呼叫函式 (Function Call)」和「宣告區域變數 (Local Variables)」視為理所當然。但實際上，這些操作背後都有一個隱形的依賴：**堆疊 (Stack)**。

讓我為你拆解為什麼我們必須在 `boot.S` 裡手動挖這塊 16KB 的空間：

### 1. C 語言的「隱形前提」

C 語言的編譯器在產生機器碼時，預設 CPU 已經有一個準備好的「堆疊空間」。
當你呼叫 `kernel_main()` 或是任何一個 C 函式時，背後會發生這些事：

* **儲存返回位址：** 系統要知道函式執行完後，該回到哪一行程式碼繼續往下跑。
* **傳遞參數：** 如果函式有參數（例如 `void foo(int a, int b)`），這些參數會被塞進堆疊裡。
* **區域變數：** 函式裡面宣告的變數（例如我們 VGA 驅動裡的 `size_t index = 0;`），也是存放在堆疊裡。

在一般的作業系統（如 Linux）中，OS 會在啟動你的應用程式前，幫你切好這塊空間。但現在**我們自己就是作業系統**，系統剛開機時是一片荒蕪，如果我們沒有手動設定好堆疊就直接跳進 C 語言，C 語言一嘗試儲存變數，就會寫入未知的隨機記憶體位址，導致系統瞬間崩潰（Triple Fault 重啟）。

### 2. x86 架構的「向下生長」特性

這也是一個有趣的硬體設計。在 x86 架構中，堆疊是**由高位址往低位址生長 (Grows downwards)** 的。

我們來看看這段組合語言：

```assembly
section .bss      ; BSS 區段：用來存放「未初始化」的資料
align 16          ; 記憶體對齊，確保效能與穩定性
stack_bottom:     ; 標籤：堆疊的最底層（低位址）
    resb 16384    ; 保留 (Reserve Bytes) 16384 bytes = 16 KB 的空間
stack_top:        ; 標籤：堆疊的最頂層（高位址）

```

因為堆疊是往下長的，所以我們必須把 CPU 的堆疊指標暫存器（`esp`，Extended Stack Pointer）指向這塊空間的**最高位址**，也就是 `stack_top`：

```assembly
mov esp, stack_top

```

當 C 語言把一個變數 `push` 進堆疊時，`esp` 的位址就會減小，慢慢往 `stack_bottom` 的方向靠近。16KB 對於目前的早期核心來說，已經非常足夠應付很深的函式呼叫了。

### 3. 為什麼放在 `.bss` 區段？

`.bss` (Block Started by Symbol) 區段專門用來放全域且未初始化的變數。
如果我們把它放在 `.data` 區段，這 16KB 的全零資料就會被實實在在地寫入 `myos.bin` 這個檔案裡，白白浪費硬碟空間。放在 `.bss` 區段，檔案大小不會增加，而 Bootloader (GRUB) 在把核心載入記憶體時，會聰明地自動在 RAM 裡面劃出這 16KB 並填滿 0。

---

簡單來說：**這塊 16KB 的空間，就是我們為了迎接 C 語言這個「高級外賓」，預先鋪好的紅地毯。** 有了它，C 語言才能安穩地執行函式呼叫跟宣告變數。
