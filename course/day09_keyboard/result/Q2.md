
# Q2: 我想把 assembly 和 c lib 分別放在 asm lib 這兩個目錄，需要做怎樣的調整？

# A

這是一個非常棒的決定！將不同職責的程式碼歸類到專屬的資料夾，正是從「實驗性專案」邁向「工程級系統」的標準起手式。就像 Linux 核心原始碼也會把架構相關的組合語言放在 `arch/`，通用功能放在 `kernel/` 或 `lib/` 一樣。

要把結構拆分成 `asm/` 和 `lib/`，我們主要需要調整 **Makefile**（告訴編譯器去哪裡找檔案）以及編譯參數（確保 C 語言找得到 `.h` 標頭檔）。

### 步驟 1：建立新的目錄結構

請先在專案根目錄下建立 `asm` 與 `lib` 資料夾，並把檔案搬移進去。你的目錄結構看起來會像這樣：

```text
.
├── asm/
│   ├── boot.S
│   ├── gdt_flush.S
│   └── interrupts.S
├── lib/
│   ├── gdt.c / gdt.h
│   ├── idt.c / idt.h
│   ├── io.h
│   ├── keyboard.c / keyboard.h
│   ├── kernel.c
│   ├── tty.c / tty.h
│   └── utils.c / utils.h
├── Dockerfile
├── grub.cfg
├── linker.ld
└── Makefile

```

### 步驟 2：修改 Makefile

因為檔案被移進了子目錄，我們必須更新 Makefile 裡的 `wildcard` 搜尋路徑。另外，為了確保 `boot.o` (現在變成 `asm/boot.o`) 依然排在連結清單的最前面，我們也要微調變數。

請將你的 `Makefile` 更新為以下內容：

```makefile
DOCKER_CMD = docker run --platform linux/amd64 --rm -v $(PWD):/workspace -w /workspace os-builder

# [新增] -Ilib 告訴 GCC：如果遇到 #include "xxx.h"，請去 lib/ 目錄下找！
CFLAGS = -m32 -ffreestanding -O2 -Wall -Wextra -Ilib

# [修改] 讓 wildcard 去子目錄尋找檔案
C_SOURCES = $(wildcard lib/*.c)
C_OBJS = $(C_SOURCES:.c=.o)

ASM_SOURCES = $(wildcard asm/*.S)
ASM_OBJS = $(ASM_SOURCES:.S=.o)

# [修改] 把 boot.o 的路徑改成 asm/boot.o
OBJS = asm/boot.o $(filter-out asm/boot.o, $(ASM_OBJS)) $(C_OBJS)

all: build-env myos.iso

build-env:
	docker build -t os-builder .

myos.iso: myos.bin grub.cfg
	mkdir -p isodir/boot/grub
	cp myos.bin isodir/boot/myos.bin
	cp grub.cfg isodir/boot/grub/grub.cfg
	$(DOCKER_CMD) grub-mkrescue -o myos.iso isodir
	rm -rf isodir

myos.bin: $(OBJS)
	$(DOCKER_CMD) ld -m elf_i386 -n -T linker.ld -o myos.bin $(OBJS)

# 編譯規則不需要改，因為 $@ 會自動帶入 lib/xxx.o，$< 會帶入 lib/xxx.c
%.o: %.c
	$(DOCKER_CMD) gcc $(CFLAGS) -c $< -o $@

%.o: %.S
	$(DOCKER_CMD) nasm -f elf32 $< -o $@

# [修改] 清理指令要一併刪除子目錄裡產生的 .o 檔
clean:
	rm -f asm/*.o lib/*.o *.bin *.iso
	rm -rf isodir

```

### 關於 C 語言的 `#include` 調整

因為我們在 Makefile 的 `CFLAGS` 中加入了 `-Ilib` 這個強大的參數，它等於把 `lib/` 加入了系統的預設搜尋路徑。
所以，即使所有的 `.c` 檔案都位於 `lib/` 底下，你**完全不需要**修改程式碼裡的 `#include` 寫法。

原來的：

```c
#include "tty.h"
#include "utils.h"

```

會繼續完美運作，GCC 在編譯時會自動去 `lib/tty.h` 找到它們。這對於未來就算你想把 `utils.c` 再移到 `lib/utils/` 裡面，只要多加一個 `-Ilib/utils` 就能無痛轉移！

### 執行與驗證

搬移檔案並更新 Makefile 後，請執行：

```bash
make clean
make

```

你可以觀察一下終端機印出來的編譯指令，它會變成類似：
`gcc ... -c lib/kernel.c -o lib/kernel.o`
`nasm ... asm/boot.S -o asm/boot.o`

這代表你的自動化建置系統已經成功升級，完美支援多層級目錄架構了！

這是一個非常棒的推論！不過這次的兇手**不是** `boot.S`，你的 `boot.S` 寫得完全正確，不需要去修改 `call kernel_main` 的指向。在 C 語言與組合語言的編譯世界裡，不管檔案放在哪個資料夾，函式的「符號名稱 (Symbol Name)」都不會改變。

我們來當一下偵探，仔細看你貼上的 Log，特別是最後執行 `ld` (連結器) 的那一行：

```bash
docker run ... ld -m elf_i386 ... -o myos.bin asm/boot.o asm/gdt_flush.o asm/interrupts.o lib/gdt.o lib/idt.o lib/keyboard.o lib/tty.o lib/utils.o

```

你有發現少了一個最重要的大人物嗎？**`lib/kernel.o` 離奇失蹤了！**

而且在上面 GCC 編譯的過程中，也沒有看到 `gcc ... -c lib/kernel.c` 的動作。這就是為什麼 Linker 會抱怨 `undefined reference to 'kernel_main'`（找不到 `kernel_main` 的定義），因為包含這個函式的檔案根本沒有被打包進去！

### 🔍 解決方案：檢查檔案位置

既然我們的 `Makefile` 是用 `wildcard lib/*.c` 自動抓取檔案，這通常代表只有一個可能：
**`kernel.c` 沒有被正確地放進 `lib/` 資料夾裡面！** 它可能還遺留在專案的根目錄，或者不小心被更名了。

請做以下兩個步驟：

1. 檢查一下目錄結構，把 `kernel.c` 確實移動到 `lib/` 資料夾底下。
2. 確保它的檔名是正確的 `kernel.c`。

處理好之後，直接再次執行：

```bash
make clean
make

```

這次你應該就會在編譯過程中看到 `lib/kernel.o` 被成功編譯，並且順利通過最後的 Linking 階段了！
