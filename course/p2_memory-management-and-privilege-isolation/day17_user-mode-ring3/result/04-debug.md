# A

這份 Log 實在太有價值了！我要先恭喜你，雖然畫面還是重啟，但**你的 CPU 已經成功墜入凡間了！**

我們再來當一次底層偵探，看看這份 Log 揭露了什麼驚人的事實：

### 🔍 案發現場解密：成功降級與分頁權限衝突

請看你貼的 Log 倒數幾行的這段致命紀錄：
`check_exception old: 0xffffffff new 0xe`
`0: v=0e e=0005 i=0 cpl=3 IP=001b:001003d0 pc=001003d0 SP=0023:00106fdc CR2=001003d0`

這裡藏著三個巨大的情報：

1. **`cpl=3` (Current Privilege Level = 3)：** 太神啦！這證明我們的 `iret` 偷天換日魔法完全生效，CPU 確確實實已經切換到了特權最低的 User Mode！
2. **`v=0e` (Exception 14 = Page Fault)：** 系統死於分頁錯誤（虛擬記憶體違規）。
3. **`e=0005` (Error Code 5)：** 把 `5` 轉成二進位是 `0101`。
* 第 0 位 (1): 記憶體分頁是存在的 (Present)。
* 第 1 位 (0): 這是一次讀取/執行操作 (Read/Execute)。
* 第 2 位 (1): **這是一次來自 User Mode (Ring 3) 的存取！**



**💡 破案總結：**
CPU 成功降級到了 Ring 3，並且試圖去執行你的 `my_first_user_app` 函式（位址在 `0x001003d0`）。
但是！回想一下我們在 **Day 10 (Paging)** 設定的分頁字典。我們當初把 0MB~4MB 的第一張分頁表權限設為 `3` (`011` 二進位，代表 Present + Read/Write)。
在 x86 的分頁機制中，**第 2 個 bit (數值為 4) 是 User/Supervisor 旗標**。如果沒有加上 `4`，這塊記憶體就是「上帝專屬（Ring 0~2）」，平民（Ring 3）連看一眼都會被警衛當場擊斃！

因為我們的 `my_first_user_app` 函式是跟 Kernel 綁在一起編譯的（都在前 4MB 的記憶體空間裡），所以當平民身分的 CPU 試圖去讀取前 4MB 的程式碼時，當場觸發 Page Fault 當機。

---

### 🛠️ 解決方案：開放 0~4MB 的平民權限

在一個真正的現代 OS 中，Kernel 的記憶體（Ring 0）和 User App 的記憶體（Ring 3）會被嚴格分開在不同的分頁。但因為我們現在是把「假 App」寫在 Kernel 內部測試，所以我們必須暫時把前 4MB 空間也開放給平民讀寫。

請打開 **`lib/paging.c`**，把 `init_paging` 裡面的第一張表（前 4MB）權限從 `3` 改成 `7`（`3` + `4` User 旗標 = `7`）：

```c
void init_paging(void) {
    // ... 略 ...

    // 初始化第一張表 (1:1 對應 0MB~4MB)
    for(int i = 0; i < 1024; i++) {
        // [修改] 將權限從 3 改成 7 (Present | Read/Write | User)
        first_page_table[i] = (i * 0x1000) | 7; 
    }
    
    // ... 略 ...

    // [修改] 目錄的第 0 項也必須加上 User 權限 (7)
    page_directory[0] = ((uint32_t)first_page_table) | 7;
    
    // 其他目錄項 (2GB, 3GB) 先維持 3 不變，因為 User App 不需要碰到那裡
    page_directory[512] = ((uint32_t)second_page_table) | 3;
    page_directory[768] = ((uint32_t)third_page_table) | 3; 

    // ... 略 ...
}

```

改好之後，重新 `make clean && make`。這一次，平民（Ring 3）終於有權利去執行存在前 4MB 空間裡面的 `my_first_user_app` 了！
