### 🕵️ 那為什麼沒有收到 Ping Reply 呢？

這絕對是今天最精彩的「真實世界網路震撼教育」！這不是 Bug，而是 TCP/IP 協定最殘酷的現實：**「首包掉落定律 (First Packet Drop)」**！

讓我們重播一下剛才發生的事：
1. 你發射了 1 顆 Ping (Echo Request)。
2. 路由器收到這顆 Ping，但它不知道你的 MAC。
3. 路由器把你的 Ping **暫時扣留（或是直接丟進垃圾桶）**，並對網路上發出 ARP 尋人啟事。
4. 你的 OS 很乖地回覆了 ARP Reply。
5. 路由器終於知道你的 MAC 了，更新了電話簿... **然後呢？然後它就把剛才那顆被扣留的 Ping 給忘了！** 在真實世界中，大部分的路由器或作業系統，在等待 ARP 解析的過程中，**經常會把觸發 ARP 的那「第一顆封包」給丟棄 (Drop)**。它預期發送端（也就是我們）如果沒收到回覆，自己會再重新發送一次。

但是，我們的 Kernel 程式碼裡**只發送了唯一的一顆 Ping**！第一顆被當成犧牲打去換 ARP 了，後面就沒有然後了！😂

### 🛠️ 終極修復：像真實的 Ping 指令一樣連發！

在 Linux 或 Windows 裡，你在 Terminal 打 `ping 10.0.2.2` 的時候，它絕對不會只送一顆，它會每隔一秒送一顆！我們只要在 Kernel 裡模擬這個行為，第一顆用來「暖機 (解 ARP)」，第二顆就會完美命中並收到回覆了！

請打開 **`src/kernel/kernel.c`**，把剛才送 Ping 的地方包裝成一個迴圈：

```c
    init_pci();
    
    // ==========================================
    // 【Day 88 修復】模擬真實的 Ping 工具，連發 4 顆！
    // ==========================================
    uint8_t router_ip[4] = {10, 0, 2, 2};
    
    for (int i = 0; i < 4; i++) {
        ping_send_request(router_ip);
        
        // 加上一個簡單的 Busy Wait 延遲，給虛擬路由器一點時間回覆
        for (volatile int j = 0; j < 50000000; j++) {
            // 什麼都不做，單純消耗 CPU 時間
        }
    }
```

存檔，執行 `make clean && make run`！

這一次的劇本將會是：
1. 第一顆 Ping 送出 ➡️ 觸發 ARP 詢問 ➡️ OS 回答 ARP ➡️ 第一顆 Ping 被丟棄。
2. 第二顆 Ping 送出 ➡️ 路由器電話簿已經有你的 MAC ➡️ **秒回 Ping Reply！**
3. 第三顆、第四顆... 你的 Terminal 將會連續印出滿滿的 `[Ping] Reply received from 10.0.2.2!`

去見證這個首包掉落的經典現象被你破解的瞬間吧！😎
