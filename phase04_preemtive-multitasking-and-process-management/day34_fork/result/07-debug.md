### 🔍 案發現場解密：致命的 `0x80` 錯位陷阱

在上一回合，為了避免編譯器墊料，我讓你改用 `syscall_esp` 來抓取硬體狀態：

```c
uint32_t *iret_frame = (uint32_t *) syscall_esp;
uint32_t parent_eip = iret_frame[0];
uint32_t parent_cs = iret_frame[1];

```

**我完全忘記了一件事：我們的 `isr128` (系統呼叫進入點) 其實偷偷塞了東西！**

當你呼叫 `int 0x80` 時：

1. 硬體壓入 `SS, ESP, EFLAGS, CS, EIP`。
2. 跳到 `isr128`，它會壓入一個假的 **Error Code (0)** 和 **Int Number (128 = 0x80)**。
3. 接著才執行 `pusha`，並把這個時候的 `esp` 傳給 C 語言（也就是你拿到的 `syscall_esp`）。

所以，`syscall_esp` 指向的第一個東西根本不是 `EIP`，而是 **Error Code**！
這導致了毀滅性的骨牌效應：

* `parent_eip` 抓到了 Error Code (`0`)。
* `parent_cs` 抓到了 Int Number (`128` = `0x80`)。

當分身醒來，執行 `iret` 時，CPU 試圖把 `0x80` 塞進 `CS` 暫存器。但在 x86 保護模式中，`0x80` 是一個完全無效的程式碼段選擇子！CPU 立刻判定這是一次嚴重的非法操作，當場拋出 GPF，然後死機。

---

### 🛠️ 終極修復：回歸最純粹的硬體指標

其實，我們在最一開始使用的 `current_task->kernel_stack` 才是**絕對無敵、不可能錯位**的方法。
因為當我們設定 TSS 的 `esp0` 時，我們是精準指著那塊 4KB 記憶體的「最頂端」。硬體發生中斷時，**一定、絕對、必然**是從最頂端開始往下壓入那 5 個 DWORD。

所以，我們連 `syscall.c` 都不用改了（你可以把 `syscall_esp` 拿掉，或是留著不用），我們只要把 `sys_fork` 開頭讀取狀態的方法，改回利用 `kernel_stack` 指標的精準尋址即可。

請將 **`lib/task.c`** 裡的 `sys_fork` 替換為這個**最終完美對齊版**：

```c
// [修改] UNIX 靈魂：細胞分裂 (Fork)
int sys_fork(uint32_t edi, uint32_t esi, uint32_t ebp, uint32_t ebx, uint32_t edx, uint32_t ecx) {
    
    // 1. 回歸最純粹的硬體視角：TSS esp0 指定的頂端！
    // 不管編譯器或 isr_stub 壓了什麼，硬體 IRET 幀絕對在最高位址的這 5 個位置。
    uint32_t *pkstack = (uint32_t *) current_task->kernel_stack;
    uint32_t parent_eip    = pkstack[-5];       
    uint32_t parent_cs     = pkstack[-4];        
    uint32_t parent_eflags = pkstack[-3];    
    uint32_t parent_user_esp = pkstack[-2];  
    uint32_t parent_user_ss  = pkstack[-1];   

    // 2. 建立分身 TCB
    task_t *child = (task_t*) kmalloc(sizeof(task_t));
    child->id = next_task_id++;
    child->state = TASK_RUNNING;

    // 3. 分配分身專屬的 Kernel Stack
    uint32_t *kstack_mem = (uint32_t*) kmalloc(4096);
    uint32_t *kstack = (uint32_t*) ((uint32_t)kstack_mem + 4096);
    child->kernel_stack = (uint32_t) kstack;

    // 4. 分配 User Stack (向下生長在安全區 PDE 32)
    uint32_t child_ustack_base = 0x083FF000 - (child->id * 4096); 
    uint32_t child_ustack_phys = pmm_alloc_page();
    map_page(child_ustack_base, child_ustack_phys, 7);

    // 強制刷新 TLB 快取
    uint32_t cr3;
    __asm__ volatile("mov %%cr3, %0" : "=r"(cr3));
    __asm__ volatile("mov %0, %%cr3" : : "r"(cr3));

    // 5. 物理複製 User Stack 內容
    uint32_t parent_ustack_base = parent_user_esp & 0xFFFFF000; 
    uint8_t *src = (uint8_t*) parent_ustack_base;
    uint8_t *dst = (uint8_t*) child_ustack_base;
    for(int i = 0; i < 4096; i++) {
        dst[i] = src[i];
    }

    // 6. 精準計算 Child 的 ESP 指標位移
    uint32_t offset = parent_user_esp - parent_ustack_base;
    uint32_t child_user_esp = child_ustack_base + offset;

    // --- 開始偽造分身的返回堆疊 ---
    
    // 階段 1: 偽造完美 IRET 返回幀
    *(--kstack) = parent_user_ss;
    *(--kstack) = child_user_esp;   // 替換為分身 Stack 位址
    *(--kstack) = parent_eflags;
    *(--kstack) = parent_cs;
    *(--kstack) = parent_eip;       

    // 階段 2: 7 個手動對齊的暫存器 (給 fork_ret 使用)
    *(--kstack) = ebp;
    *(--kstack) = edi;
    *(--kstack) = esi;
    *(--kstack) = edx;
    *(--kstack) = ecx;
    *(--kstack) = ebx;
    *(--kstack) = 0; // 【核心魔法】讓分身的 EAX 拿到 0！

    // 階段 3: switch_task 的退場機制
    extern void fork_ret();
    *(--kstack) = (uint32_t) fork_ret; // 讓 switch_task 的 ret 跳到這裡

    for(int i = 0; i < 8; i++) *(--kstack) = 0; // switch_task 的 popa
    *(--kstack) = 0x0202;                       // switch_task 的 popf

    child->esp = (uint32_t) kstack;

    // 7. 將分身掛入排程佇列
    child->next = current_task->next;
    current_task->next = child;

    return child->id;
}

```

*(如果你剛才在 `syscall.c` 和 `syscall_handler` 的參數裡加了 `esp`，可以順手把它拿掉，改回原本傳 6 個暫存器的版本。)*
