太神了，Rick！我們遇到了作業系統開發史上最經典、最令人毛骨悚然的「奪舍事件」！👻

你提供的 Core Dump 簡直就是一份完美的法醫驗屍報告。請跟著我一起看這幾行關鍵記錄：

```text
  91: v=80 ... EAX=00000007  <-- 子行程 (echo) 呼叫 sys_exit (7)，完美結束！
  92: v=80 ... EAX=00000002  <-- 老爸 (Shell) 醒來，立刻呼叫 sys_print (2)
  93: v=80 ... EAX=00000002 EBX=f000ff53 <-- 等等！老爸傳給 sys_print 的字串指標竟然是 f000ff53！
  94: v=0e ... CR2=f000ff53  <-- Kernel 試圖印出這個位址的字串，當場 Page Fault (0x0E) 暴斃！
```

### 🔍 案發現場解密：被「奪舍」的老爸

為什麼子行程 `echo.elf` 活得好好的，完美印出了參數並結束，而老爸 `my_app.elf` 睡醒後卻像是腦袋壞掉一樣，拿著一個根本不存在的位址 (`0xf000ff53`) 去印字串？

這要回到我們昨天 `sys_exec` 中最無情的那句：「**洗腦重置**」。
因為我們現在的 Simple OS 是一個輕量級的架構，**所有的任務 (Task) 都共用同一張「分頁目錄 (Page Directory)」**。這意味著大家活在同一個虛擬記憶體宇宙裡。

1. 當初編譯時，我們在 Makefile 裡設定 `my_app.elf` 的進入點是 `-Ttext 0x08048000`。
2. 昨天我們加入 `echo.elf` 時，**我們也照抄了 `-Ttext 0x08048000`！**
3. 當 `sys_exec` 載入 `echo.elf` 時，它毫不留情地把 `0x08048000` 這個位址的記憶體給覆寫成了 `echo.elf` 的程式碼。
4. 子行程開心執行完畢死掉後，老爸 `my_app.elf` 從 `sys_wait` 醒來，試圖執行下一行指令。
5. **但是！老爸的程式碼早就被覆寫成 `echo.elf` 的形狀了！** 老爸執行了完全不屬於他的指令，產生了隨機的垃圾記憶體位址，導致 Kernel 崩潰！



這就是為什麼在現代的 Linux 或 macOS 中，作業系統必須為**每一個行程建立專屬的 Page Directory (MMU 隔離)**，讓每個程式都以為自己獨佔了 `0x08048000` 這個位址。而我們目前的架構，其實更像是早期 IoT 設備上大名鼎鼎的 **uClinux** (無 MMU 保護的 Linux)！

### 🗡️ 終極修復：搬家！

既然大家住在同一個宇宙，那最簡單的解法就是：**不要住在同一個地址！**
只要我們把 `echo.elf` 編譯到記憶體的其他區塊，它們兩個就能和平共處了。

請打開你的 **`Makefile`**，找到編譯 `echo.elf` 的那一行，把 `-Ttext 0x08048000` 改成 **`-Ttext 0x08080000`**（往後挪動大約 200KB 的距離）：

```makefile
# ... 前面不變 ...

# 【修改】連結 echo.elf，將載入位址改為 0x08080000
echo.elf: crt0.o echo.o
	@echo "==> 連結 Echo 程式..."
	$(DOCKER_CMD) ld -m elf_i386 -Ttext 0x08080000 crt0.o echo.o -o echo.elf

# ... 後面不變 ...
```

就改這一個數字！
存檔後，執行 `make clean && make run`！

這一次，`my_app.elf` 住在 `0x08048000`，`echo.elf` 被載入到 `0x08080000`，兩個程式涇渭分明。當 `echo` 執行完畢後，老爸醒來，它的程式碼依然完好如初，系統將順利回到 `SimpleOS>` 提示字元！

如果這次順利看到完美無瑕的輸出，你想試試看寫一個加法小算盤 `calc.elf` (例如輸入 `run calc 5 10` 輸出 `15`)，還是想要挑戰 OS 界最神聖的任務：為每個 Process 實作「記憶體隔離 (Memory Isolation)」呢？
