# Q: 看起來可以了，有看到兩個 user task，我發現在輸入指令的時候，會被截斷，是否因為 context switch 造成？

---

太神啦！你的直覺完全正確，這張截圖簡直是作業系統教科書裡最完美的活教材！

你看著畫面上支離破碎的指令（像是輸入 `about` 卻變成 `abo` 和 `b`），這絕對不是 Bug，這正是你的 **Context Switch (上下文切換)** 運作得太過完美，而引發的經典現象：**Race Condition (競爭危害)**！

### 🔍 案發現場解密：被撕裂的指令

我們來還原一下當你快速在鍵盤敲下 `help` 時，底層到底發生了什麼大亂鬥：

1. **共享的食物碗：** 系統裡只有一個實體的鍵盤，也只有一個 256 bytes 的環狀緩衝區 (`kbd_buffer`)。
2. **兩張嗷嗷待哺的嘴：** 你的 App 1 (Shell 1) 和 App 2 (Shell 2) 都在執行 `while(1)` 迴圈，並且都在呼叫 `sys_getchar` 緊盯著這個緩衝區。
3. **時鐘中斷的無情切割：**
* 你按下了 `h`，字元掉進緩衝區。剛好這時是 **Shell 1** 的回合，它立刻把 `h` 拿走。
* **[Context Switch 發生！]** Timer 中斷觸發，CPU 暫停了 Shell 1，切換給 **Shell 2**。
* 你按下了 `e`，字元掉進緩衝區。現在輪到 Shell 2 醒著，所以它把 `e` 拿走了！
* **[Context Switch 發生！]** CPU 又切回 **Shell 1**。
* 你按下了 `l` 和 `p`，Shell 1 拿走了 `l`，Shell 2 拿走了 `p`...



最終的結果就是：Shell 1 拿到了 `hl`，Shell 2 拿到了 `ep`。當你按下 Enter 時，兩個 Shell 各自去比對指令，當然都會印出 `Command not found`！

### 💡 這證明了什麼？

這張混亂的截圖，是底層工程師眼中的無價之寶，因為它證明了三件偉大的事：

1. **你的計時器 (PIT) 與排程器 (Scheduler) 完美運作：** 任務切換的速度極快（大約每 10 毫秒切換一次），才能把你連續敲擊鍵盤的動作切得這麼碎。
2. **TSS 與特權切換完美咬合：** CPU 在 Ring 0 與 Ring 3 之間瘋狂來回橫跳，各自使用獨立的 Kernel Stack，系統居然沒有崩潰（沒有 Page Fault 或是 Double Fault）！
3. **暴露出同步問題 (Synchronization)：** 讓你知道為什麼現代 OS 需要有 `Mutex` (互斥鎖)、`Semaphore` (號誌) 或是 `Pipe` (管線)。因為當多個行程共用資源（如鍵盤、螢幕 VGA 緩衝區）時，沒有交通警察來指揮就會天下大亂。

### 🚀 下一步：我們該怎麼辦？

這就是我們第四階段 (Phase 4) 要解決的核心問題！

在真實世界中，不可能讓兩個 Shell 同時搶同一個鍵盤。我們通常會：

1. **實作 UNIX 的 `fork` 與 `execve`：** 讓系統開機時只跑一個 Shell。當你輸入指令時，Shell 會複製自己 (`fork`)，然後用新的 App (`execve`) 來取代分身去執行任務，本尊則乖乖等待 (`wait`) 分身結束，不會同時搶鍵盤。
2. **實作焦點管理 (Focus)：** 只有在「前景 (Foreground)」的行程才能獲得讀取鍵盤的權利。
