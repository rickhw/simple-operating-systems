
### 圖解 Stack 與 User/Kernel Space 分離的概念

這張圖解釋了你昨天的思考題：**「為什麼 Ring 3 應用程式被打斷時，要強迫換一條安全的核心 Stack？」**

#### 概念解釋：為什麼需要分離？

駭客與惡意程式最喜歡攻擊的地方就是「Stack」。如果使用者空間 (Ring 3) 和核心空間 (Ring 0) 共用同一條 Stack，會有嚴重的安全隱患：

1. **資訊洩漏 (Data Leaking)：** 當使用者程式呼叫核心功能 (System Call) 或中斷發生時，核心會在 Stack 上留下一些「區域變數」或「機密資料」（例如解密金鑰）。使用者程式返回時，可以直接去讀取 Stack 裡面的殘留資料，機密不保。
2. **堆疊溢位與劫持 (Stack Overflow & Hijacking)：** 使用者程式可以故意分配一個超大的陣列把 Stack 填滿（溢位）。如果共用 Stack，使用者程式只要稍微算出距離，就能**精準覆蓋掉核心呼叫函式時存入的「返回位址」**。當核心執行完準備返回時，會直接跳到駭客預設的惡意程式碼中，駭客就徹底掌控了系統最高權限 Ring 0。

#### CPU 的硬體保護機制

為了防止這種情況，x86 硬體做了嚴格的防護：**當 CPU 在 Ring 3 執行時收到中斷，在跳到 Ring 0 的 ISR 之前，硬體會「強制要求從記憶體裡讀取一個安全、乾淨且專屬於 Ring 0 的 Stack 位址」並把 `esp` 切換過去。**

這確保了：核心執行 ISR 時使用的是硬體保證安全的紅地毯，使用者程式無法預先在上面寫入惡意資料，核心丟在上面的資料使用者也讀不到。
