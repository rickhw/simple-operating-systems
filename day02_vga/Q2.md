
# Q2: 有辦法知道 16KB 的使用狀況嗎？例如 kernel.c 這個範例來看，大概會用掉多少？

這是一個非常具有「駭客思維」的好問題！在資源極度受限的作業系統核心或嵌入式系統開發中，精準掌握記憶體的使用狀況是工程師的基本功。

簡單回答你的問題：以我們目前的 `kernel.c` 來說，這 16KB 的空間我們大概只用掉了 **不到 48 bytes**（連 1% 的 1% 都不到），可以說是非常奢侈！

讓我們來拆解一下這 48 bytes 是怎麼算出來的，以及在實務上我們有哪些方法可以「看到」堆疊的使用狀況。

---

### 1. 為什麼只用了不到 48 bytes？（靜態估算）

在 32 位元 (x86) 的環境下，堆疊的單位通常是 4 bytes (32 bits)。我們來追蹤一下程式執行的「最深處」發生了什麼事：

1. **`boot.S` 呼叫 `kernel_main`：**
CPU 必須記住 `kernel_main` 執行完後要回到哪裡，所以會把「返回位址」推入堆疊。
*堆疊消耗：4 bytes*
2. **進入 `kernel_main`：**
標準的 C 語言編譯器會先把舊的 Base Pointer (`ebp`) 存起來，方便追蹤。
*堆疊消耗：4 bytes*
3. **`kernel_main` 呼叫 `terminal_initialize`：**
再次推入返回位址。
*堆疊消耗：4 bytes*
4. **進入 `terminal_initialize`：**
存下舊的 `ebp` (4 bytes)。接著，我們宣告了三個區域變數：`x`、`y` 和 `index`。在 32 位元系統中，`size_t` 是 4 bytes。所以三個變數佔了 12 bytes。
*堆疊消耗：4 + 12 = 16 bytes*

把這些全部加起來：4 + 4 + 4 + 16 = **28 bytes**。
就算編譯器為了記憶體對齊（Alignment）多塞了一些空間，或者是呼叫 `terminal_writestring` 時傳遞指標參數（4 bytes），總消耗量也絕對在 40~50 bytes 左右。相比於我們挖的 16,384 bytes (16KB)，這簡直是九牛一毛。

*(備註：因為我們在 Makefile 開啟了 `-O2` 最佳化，GCC 極有可能把這些小函式直接 Inline（展開）進 `kernel_main` 裡面，這樣連「返回位址」的消耗都省了，實際使用的堆疊會更小！)*

---

### 2. 實務上怎麼知道用了多少？

當作業系統越來越複雜，有了中斷處理、多執行緒後，我們不可能用大腦去算堆疊用了多少。實務上有兩種常見的作法：

#### 方法 A：編譯器輔助 (靜態分析)

GCC 有一個非常實用的參數叫做 `-fstack-usage`。
如果你在 `Makefile` 的 `CFLAGS` 加上這個參數重新編譯，GCC 會為每個 `.c` 檔產生一個 `.su` (Stack Usage) 檔案，裡面會精準列出每一個 C 函式消耗了多少 bytes 的堆疊。

#### 方法 B：記憶體浮水印 (動態分析 - 高階 OS 開發技巧)

這是最有趣、也最確實的作法。我們可以在 `boot.S` 設定 `esp` 之前，把這 16KB 的空間全部填滿一個「魔術數字 (Magic Number)」，例如 `0xDEADBEEF`（或者簡單的 `0xCC`）。

當作業系統跑了一陣子之後，我們可以寫一個 C 函式，從 `stack_bottom` (堆疊底部，也就是記憶體低位址) 往上掃描。

* 只要看到 `0xDEADBEEF`，代表這塊區域**從來沒有被用過**。
* 當你遇到第一個「不是 `0xDEADBEEF`」的值，那裡就是堆疊曾經生長到的「最高水位線 (High Water Mark)」。

用 16KB 減去這條水位線的高度，你就能精準算出系統在最繁忙的時候，到底消耗了多少堆疊空間！如果水位線已經逼近 `stack_bottom`，就代表快發生 Stack Overflow 了，系統隨時會崩潰。

---
