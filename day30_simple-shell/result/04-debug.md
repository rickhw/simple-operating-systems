# Q: 跑起來，有看到 SimpleOS> 的提示字，不過鍵盤不管敲什麼都沒反應

這真的是作業系統開發中最經典、最讓人抓狂、但也最迷人的一個「神級 Bug」！

你看著畫面上完美的 `SimpleOS>` 提示字元，滿心歡喜地敲打鍵盤，結果系統卻像是一灘死水，毫無反應。這不是你的程式碼寫錯了，而是你正式觸發了現代作業系統中最棘手的問題：**核心死結 (Kernel Deadlock)！**

### 🔍 案發現場解密：為什麼鍵盤會失效？

我們來推理一下當你敲下指令時，CPU 到底經歷了什麼：

1. Shell (`app.c`) 呼叫了 `sys_getchar()`。
2. 觸發 `int 0x80`，CPU 從 Ring 3 切換回 Ring 0 (Kernel)。
3. **【致命盲點】**：在 x86 架構中，當 CPU 透過「中斷門 (Interrupt Gate)」進入中斷處理程式（例如我們的 Syscall Handler）時，它會**自動關閉所有硬體中斷**（也就是把 EFLAGS 暫存器裡的 `IF` 旗標設為 0），以防止中斷被另一個中斷打斷。
4. 接著，你的程式碼走進了 `keyboard_getchar()`，看到緩衝區是空的，於是執行了 `__asm__ volatile("hlt");`，讓 CPU 進入休眠，等待下一次按鍵。
5. **死結誕生！** CPU 正在休眠等待鍵盤發出中斷，但是**你剛才（被動地）把所有硬體中斷都遮蔽了！** 於是，鍵盤發出的訊號被 CPU 拒絕於門外，整個作業系統就這樣永遠地睡著了。

---

### 🛠️ 終極修復：在 Syscall 中「解除封印」

要打破這個死結非常簡單，我們只需要在等待按鍵之前，主動告訴 CPU：「我現在準備好接收硬體中斷了，請把封印解除！」

請打開你的 **`lib/keyboard.c`**，把 `keyboard_getchar` 修改成這樣：

```c
// 讓 Kernel 或 Syscall 讀取字元
char keyboard_getchar() {
    // 如果緩衝區是空的，就讓 CPU 進入休眠 (hlt)，直到下一次中斷 (按鍵) 喚醒它！
    while (kbd_head == kbd_tail) {
        // 【關鍵修復】解除中斷封印！
        // Syscall 預設會關閉中斷，我們必須手動開啟 (sti)，否則 CPU 會永遠睡死在 hlt！
        __asm__ volatile("sti"); 
        
        // 進入休眠，等待按鍵中斷喚醒
        __asm__ volatile("hlt"); 
    }

    char c = kbd_buffer[kbd_tail];
    kbd_tail = (kbd_tail + 1) % KBD_BUFFER_SIZE;
    return c;
}

```

加上這一行 `__asm__ volatile("sti");` 後，當 CPU 執行 `hlt` 休眠時，鍵盤的訊號就能夠成功傳達給 CPU，喚醒它執行 `keyboard_handler`，把字元塞進 Buffer 裡，然後打破迴圈！

*(💡 註：如果在 Day 9 時，你的架構中有使用 `init_keyboard()` 函式來向 IDT 註冊中斷，請確認你有在 `kernel.c` 的 `kernel_main` 裡呼叫它喔！如果你的 `isr33` 是直接 hardcode 呼叫 `keyboard_handler`，那就不需要。)*

### 🔍 截圖裡的隱藏亮點：非同步中斷的證明

不知道你有沒有注意到截圖裡的一個有趣細節？
當你打字打到一半時，畫面上突然冒出了 `Uptime: 1 seconds` 和 `Uptime: 2 seconds`，把你的字給截斷了。

這可不是 Bug，這是**最完美的 Concurrency（並行處理）證明**！
這代表我們在 Day 11 寫的 PIT（時鐘中斷）依然在背景精準地跳動。當你的 Ring 3 應用程式正在等待鍵盤輸入時，硬體時鐘每秒鐘準時觸發 `IRQ 0`，強行把 CPU 從 Ring 3 抓回 Ring 0，執行 `timer_handler` 印出時間，然後再毫髮無傷地切換回 Ring 3 讓你繼續打字。

這就是現代作業系統「多工作業 (Multitasking)」的最底層原理！
*(💡 如果你覺得時間一直跳出來很干擾打字，只要去 `lib/timer.c` 裡面把印出 Uptime 的 `kprintf` 註解掉，重新 `make run` 世界就清靜了～)*

回顧這一個月，你獨自一人走過了計算機科學最陡峭的幾座大山：

5. **權力下放：** 解析 ELF 執行檔，佈置平民專屬的 User Stack，完美實作 Syscall 防彈窗口，成功降級到 Ring 3。

### 🚀 接下來呢？

我們的 30 天挑戰在這裡畫下完美的句點，但 Simple OS 的生命才剛開始。未來如果你想繼續擴充它，這裡有幾個超棒的挑戰方向：

1. **實作 `ls` 指令：** 在 Kernel 裡加一個新的 Syscall (例如 Syscall 6)，讓 Shell 可以討要 SimpleFS 根目錄的檔案清單，印出像 Linux 的 `ls` 效果。
2. **支援退格鍵 (Backspace) 視覺效果：** 雖然你的 `app.c` 已經會處理倒退的邏輯，但螢幕上的字還沒消失。你可以寫一個特殊的字元處理，當收到 `\b` 時，印出「倒退、印空白、再倒退」的魔法指令。
3. **終極挑戰 - `execve`：** 寫第二個 App (例如 `echo.elf`) 放進硬碟。當你在 Shell 輸入 `echo` 時，Shell 透過 Syscall 請 Kernel 去硬碟載入 `echo.elf` 並替換掉目前的程式！
